# Autonomy Roadmap — 邁向 OpenClaw / Hermes 式自主 PM

> 狀態：路線圖（已批准方向，2026-06-16）。各階段各自有獨立 spec → plan → 實作 → dogfood 週期。
> 一次只深入設計一個階段，後階段依前階段學到的東西再定案。

## 背景與動機

使用者希望 agency-dashboard 的 PM（編排者）更像 OpenClaw / Hermes 那種「丟個目標，它自己持續做下去」的自主 agent。並具體提到：透過 PM 送出工作後，PM 還能繼續；以及不清楚「不同 PM 各負責什麼」。

### 現況診斷（基於 code，2026-06-16）

- **只有一個 PM**：真正接到 orchestrator 的只有 `agents-orchestrator`（`server/src/routes/sessions.ts:323-378`）。catalog 裡的 manager 型 agent（高級項目經理、產品經理、幕僚長、工作流架構師、項目牧羊人…）都只是可被派工的普通 agent，沒有任何一個被接成「PM」。→ 使用者「不知道不同 PM 負責什麼」是因為**這個分工目前不存在**。
- **PM 派工後已半能繼續**：`execute` 模式非阻塞（`dispatchRunner.ts:124-138`、結果用 `[[EXEC_REPORT]]` 推回 PM，`sessions.ts:403-407`）；`consult` 模式同步等（240s）。**真正缺的是「自走」**——回報推回來後 PM 不會自己決定下一步，停在那等使用者開口。
- **已有一個自主迴圈，但不是 PM**：`autonomyRunner.ts` 有分步規劃、ACTION 協定、批准閘、步數+時間預算、暫停/續跑——但 (a) 每個高風險動作都要按批准卡、(b) 是獨立的 autonomy 模式，沒跟 PM 聊天打通。等於有兩套半成品自主性各跑各的。

### 核心結論

往 OpenClaw/Hermes 不是「一個功能」，是四塊獨立能力，且有依賴順序。一次全做是錯的——後階段的設計取決於前階段 dogfood 學到的東西（呼應既有偏好：先 dogfood 2-4 週再決定下階段）。

---

## 階段一：PM 自走迴圈 + 護欄（SP2 + SP3）

**目標**：PM 派工/收到回報後，自動再跑一輪決定「下一步」，而不是停下等使用者；同時有預算護欄，低風險自動放行、高風險才攔，到頂就停下來問。**順帶把「這一個 PM 的職責與派工政策」定義清楚**，解掉角色困惑。

**主要改造（非重寫）**：
- 把 `autonomyRunner.ts` 的分步迴圈接進 PM 聊天：PM 收到 `[[EXEC_REPORT]]`/`[[CONSULT_RESULTS]]` 後自動再生一輪（自走模式開啟時）。
- 加預算護欄層：低風險（諮詢/唯讀類）自動放行；高風險（會寫東西的 execute、對外發送、花錢、破壞性）才跳批准卡；步數＋花費＋時間任一到頂 → 停下問使用者（`need_input`）。
- 自走模式的開關與目標輸入入口（PM 聊天內）。

**為何第一**：直接回應「派工後 PM 還能繼續」，且大量複用既有 code。

**詳細 spec**：待寫（本文件批准後立即進入深入設計）。

---

## 階段二：常駐主動盯事（SP4）

**目標**：PM 能有「常設目標／看守任務」——按時或按事件自己醒來、查狀態、決定要不要動手，主動回報。

**主要改造**：擴充 `scheduler.ts`（已有 cron 觸發 + 每次 fire 開新 session 的模式）；加事件觸發；定義「無人看管時允許做什麼」的政策。

**依賴**：必須先有階段一的自走+護欄，否則背景亂跑沒人擋。

---

## 階段三：多 PM 分工（SP1，可能不做）

**目標**：真的覺得一個 PM 不夠時，才拆成 2-3 個專職編排者（如 總調度／產品規劃／流程自動化），彼此能互轉派。

**判斷準則（YAGNI 守門）**：等階段一把「一個 PM 的職責」做透、使用者實際感到摩擦，再評估。在那之前不做——避免在還沒定義清楚一個 PM 時就生四個。

---

## 執行原則

1. 一階段一個 spec → plan → 實作 → dogfood，再進下一階段。
2. 後階段 spec 在前階段 dogfood 後才定案。
3. 階段三是條件式的，可能整個不做。
