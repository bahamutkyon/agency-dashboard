# 階段一 Spec：PM 自走迴圈 + 護欄

> 狀態：設計（已批准，2026-06-16）。屬 [autonomy-roadmap](./2026-06-16-autonomy-roadmap.md) 的階段一。
> 下一步：本 spec 經使用者審查後 → writing-plans 產出實作計畫。

## 1. 背景與動機

使用者要 agency-dashboard 的 PM（`agents-orchestrator`）更像 OpenClaw / Hermes：丟個目標就持續做下去，派工後 PM 還能自己接著走，而不是停下等開口。

現況（2026-06-16，基於 code）：
- PM 派工後 `execute` 已非阻塞、結果用 `[[EXEC_REPORT]]` 推回 PM，但**回報推回後 PM 不會自己決定下一步**，停在那等使用者。
- 已有一個會跑的自主迴圈 `autonomyRunner.ts`（分步規劃、ACTION 協定、步數+時間上限、暫停/續跑），但 (a) 每個高風險動作都要按批准卡、(b) 與 PM 聊天分離，兩套各跑各的。

本階段把這兩套合一：**PM 在「目標模式」下就是那個自主迴圈，外加一層自動放行護欄。**

## 2. 目標 / 非目標

**目標**
- PM 能自評「目標 vs 問題」並對目標自動進入自走迴圈。
- 自走時低風險動作自動放行、高風險才攔，到上限安全停下。
- 自走中使用者可隨時喊停或插話改方向。
- 順帶把「這一個 PM 的職責與派工政策」定義清楚（解角色困惑）。

**非目標（留後階段）**
- 背景常駐 / 自我發起工作（階段二 SP4）。
- 多個專職 PM 分工（階段三 SP1，可能不做）。
- 自動批准「對外發送 / 花錢 / 破壞性」——本階段這三類一律人工批准。

## 3. 關鍵決策（已與使用者確認）

| 決策 | 選擇 |
|---|---|
| 接法 | **做法 A**：複用 `autonomyRunner` 迴圈，改用 PM session 驅動 + 疊放行政策 |
| 放行政策 | **平衡**：諮詢/唯讀、工作區內 execute → 自動放行；對外發送、花錢、破壞性 → 人工批准 |
| 停止條件 | **多重保險**：PM 自判完成/卡住 + 步數上限 + 時間上限，任一觸發即停 |
| 啟動方式 | **依目標自動判**：PM 分類目標→自走、問題→一問一答；分類需明示且可被使用者推翻 |
| 中途控制 | **隨時停 + 插話**：停在當前步保留進度；或打字插入高優先指示進下一步 context |

## 4. 詳細設計

### 4.1 分類與觸發
PM system prompt 增補：收到使用者訊息先自評「目標（多步、需協作或跨時）」或「問題（單次可答）」，並在回覆開頭以標記明示模式：`[模式：自走]` 或 `[模式：問答]`。
- 判為目標 → server 偵測到 `[模式：自走]` 標記 → 啟動自走迴圈（建立 `AutonomyRun`）。
- 判為問題 → 維持現有一問一答路徑。
- **推翻**：使用者訊息含「直接答/不要自走」→ 強制問答；含「自走完成/你自己做完」→ 強制自走。明示指令優先於 PM 自判。

### 4.2 自走迴圈（複用 autonomyRunner，PM session 驅動）
單步循環（沿用 `autonomyRunner.loop` 結構）：
1. PM 規劃下一動作，輸出 ACTION/DISPATCH 區塊（沿用既有協定）。
2. 護欄判定（§4.3）：自動放行 → 直接執行；需批准 → 入 `PendingAction` 佇列、彈批准卡、暫停等待。
3. 執行：複用 `dispatchRunner` 的 `runConsult`（同步諮詢）/ `startExecute`（非同步外包）。
4. 結果以既有 sentinel（`[[CONSULT_RESULTS]]` / `[[EXEC_REPORT]]`）餵回 PM session。
5. PM 依結果決定：再走一步 / `goal_done` / `need_input`。
6. 回 1，直到停止條件（§4.4）。

關鍵差異 vs 現況：第 5 步後**自動再觸發 PM 一輪**（目標模式下），而非停下等使用者。

### 4.3 護欄／自動放行（平衡）
新增「放行政策表」（policy），對動作分類決定是否入待批佇列：

| 動作類別 | 對應現有 kind | 平衡政策 |
|---|---|---|
| 諮詢 / 唯讀 | dispatch(consult) | 自動放行 |
| 工作區內 execute（寫在自己工作區、不對外） | dispatch(execute) | 自動放行 |
| 對外發送（email / 訊息 / PR / 貼文） | external_send | 人工批准 |
| 花錢 | spend | 人工批准 |
| 破壞性（刪檔等） | destructive | 人工批准 |

- 自動放行的動作仍寫入 `PendingAction` 並直接標 `approved`（留稽核軌跡），不彈卡。
- 政策表為設定值，預設「平衡」；保留未來切「保守/放手/自訂」的擴充點（本階段只實作平衡 + 開關，不做完整自訂 UI）。

### 4.4 停止條件（多重保險）
任一觸發 → 迴圈停止、PM 產出進度回報、UI 顯示「續跑 / 結束」：
- **完成**：PM 輸出 `goal_done`。
- **卡住**：PM 輸出 `need_input`（缺資訊/連續失敗）。
- **步數上限**：預設 12 步（沿用 `AutonomyRun.stepCount`）。
- **時間上限**：預設 15 分鐘（沿用 `AutonomyRun.deadlineAt`）。
- 上限數字可在 UI 調整；預設值 dogfood 後校正。
- 撞上限一律安全停下，**絕不默默續跑**。

### 4.5 中途控制（隨時停 + 插話）
自走進行中，PM 聊天輸入區可：
- **停**：停在當前步、保留進度（沿用 `autonomyRunner` 的 pause）。狀態存 DB，可續。
- **插話**：使用者打字 → 標記為高優先指示，注入 PM 下一步 context（改方向/縮範圍/追加）。下一步規劃必須先消化此指示。
- 插話發生在某步執行中時：當前步跑完才套用（不中斷進行中的子 session），避免半途撕裂。

### 4.6 資料與持久化
- 沿用 `AutonomyRun`（status / stepCount / deadlineAt）記錄自走狀態。
- 新增欄位（如需）：`mode`（goal/question 判定來源）、`policy`（套用的放行政策）、`pendingInjection`（待消化的插話）。
- server 重啟續跑：沿用既有 `resumeUnfinishedRuns()` 模式。
- 自走產生的子任務照舊開真 session（標題前綴沿用「🤝 受派諮詢 / 🛠️ 外包執行」），自動進學習捕捉、從主歷史隱藏。

### 4.7 UI
- 把 `AutonomyPanel` 從獨立模式收進 PM 聊天視窗：自走時於對話流顯示「步驟進度（n/上限）+ 當前動作摘要 + 停/插話控制」。
- 高風險動作仍彈現有 `ActionApprovalCard`。
- 自走結束（完成/卡住/上限）顯示回報卡 + 「續跑 / 結束」按鈕。
- `dispatch:done` 等 socket 事件沿用。

### 4.8 錯誤處理
- 單步子任務失敗 → 錯誤回饋給 PM，PM 可重試或換路（沿用 consult 容錯）。
- 子任務逾時 → 回部分結果 + `status: timeout`（沿用現有 240s/30min 機制）。
- 連續 N 步失敗（預設 3）→ 視為卡住，觸發 `need_input` 停下。
- 撞任何上限 → 安全停下回報。

### 4.9 測試
- HTTP 端點測試沿用 `server/src/app.test.ts` 模式（ephemeral 埠 + fetch）。
- 迴圈單元測試：分類判定、政策放行/攔截矩陣、三種上限觸發、停+插話、續跑、連續失敗轉 need_input。
- 維持 server `npm test = tsc --noEmit && vitest` 型別關卡、client `tsc -b` 零錯誤。

## 5. 風險與待 dogfood 校正
- 分類誤判率（目標↔問題）：靠明示標記 + 可推翻 + 護欄兜底降低衝擊；dogfood 觀察誤判頻率。
- 上限預設值（12 步 / 15 分）為猜測，需 dogfood 校。
- 自動放行「工作區內 execute」的邊界定義（何謂「不對外」）需在實作時明確界定，避免誤判把對外動作當工作區內。
